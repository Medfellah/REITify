Project: REITify — REIT 10-K analyzer
Purpose & intent
Real estate analysts at firms like Starwood Capital spend 3-4 hours manually extracting key data points from each REIT 10-K filing. The goal of reitify is to reduce that to under 5 minutes by using an LLM to locate, extract, and cite the exact passages an analyst needs — so they can move faster without sacrificing trust in the output.
The tool is built for a single analyst persona (Jamie) doing early-stage REIT screening. She needs to pull 4 specific data points per company, across 6 companies this quarter. Every output must be traceable back to the source filing — she's putting numbers in investment memos, not blog posts.
MVP scope
This is a v1 focused on doing one thing well: reliable extraction with citations. No auth, no database, no history, no multi-company comparison. One URL in, four structured outputs out. Everything else is a future version.
What's intentionally cut from v1:

PDF filing support (HTML only)
Multi-company batch processing
Historical comparison (this year vs last year)
Risk flagging or anomaly detection
Saved sessions or export history
Support for non-REIT 10-K filings

What it does
User pastes a SEC EDGAR 10-K URL (HTML format). The app fetches the filing, chunks it by section, runs 4 targeted LLM extractions, and returns structured results — each with a verbatim source quote and section reference from the filing.
The 4 extractions:

Tenant concentration — top tenants by % of annualized base rent
Geographic exposure — % breakdown by region
Debt maturity schedule — amounts due by year
Material lease expirations in the next 24 months

Test URL
https://www.sec.gov/Archives/edgar/data/0001045609/000095017025021272/pld-20241231.htm
Where the LLM is used
Exactly 4 LLM calls — one per extraction. Each call receives a targeted chunk of the filing (not the full document) and a specific prompt asking for the data point plus a verbatim source quote from the text. No LLM is used for routing, formatting, UI logic, or anything else. The model is Claude claude-sonnet-4-5 (Anthropic), hardcoded — no model selector needed in v1.
Why citations matter
The reliability problem with LLM extraction is hallucination — the model confidently returns a number that isn't in the filing. The mitigation in reitify is mandatory source citations: every extraction must return the exact passage it drew from, displayed to the user. This means Jamie can verify any number in under 10 seconds. Without citations, the tool is a liability. With them, it's trustworthy enough for memo use.
Stack
Pick the simplest modern stack that deploys cleanly to Vercel. No database, no auth. Single environment variable: ANTHROPIC_API_KEY.
UI

Clean, minimal, professional — finance tool aesthetic

Loading state: results appear card by card as each extraction completes — don't wait for all 4

## How the single API endpoint works

One POST to `/api/analyze`. Server-Sent Events stream back 4 events — one per extraction — as each finishes. The client renders each card the moment its event arrives.

```
client POST /api/analyze  ─────────────────────────────▶  server
                                                            │
                          ◀── SSE: { id:0, status:'done' } ─┤  (tenant)
                          ◀── SSE: { id:1, status:'done' } ─┤  (geo)
                          ◀── SSE: { id:2, status:'done' } ─┤  (debt)
                          ◀── SSE: { id:3, status:'done' } ─┘  (leases)
```

Server processes them sequentially. This avoids fetching the same ~5MB HTML file 4 times, and the user still sees progressive results.

## Chunking approach (the important part)

A PLD 10-K HTML file is roughly 3–6 MB of raw HTML — far too large to stuff into a single LLM call even with a 200k context window, and doing so would dilute focus and burn unnecessary tokens. The solution is **section-aware chunking**: find the specific section of the filing that contains each data point, and pass only that.

### Step 1 — Fetch and clean

Fetch the HTML from SEC EDGAR. Strip `<script>`, `<style>`, navigation, and header/footer boilerplate. Preserve table structure — the financial data we need almost always lives in HTML tables (`<table>`, `<tr>`, `<td>`), and those need to survive the parse.

### Step 2 — Section boundary detection

SEC 10-K filings follow a mandated structure. The HTML will have headings or bolded text matching patterns like:

```
Item 1.   Business
Item 1A.  Risk Factors
Item 2.   Properties
Item 7.   Management's Discussion and Analysis
Item 8.   Financial Statements and Supplementary Data
```

The chunker scans the cleaned text for these Item-level boundaries using a regex (`/Item\s+\d+[A-Z]?\b/i`) and splits the document into named sections. This gives us a section map:

```ts
[
  { name: "Item 1 - Business",      content: "..." },
  { name: "Item 2 - Properties",    content: "..." },
  { name: "Item 7 - MD&A",          content: "..." },
  { name: "Item 8 - Financial Stmts", content: "..." },
  ...
]
```

### Step 3 — Extraction-to-section mapping

Each of the 4 data points maps to known sections of a 10-K:

| Data point | Primary section(s) | Why |
|---|---|---|
| Tenant concentration | Item 1 (Business) | Tenant schedules, top-10 lists live here |
| Geographic exposure | Item 1 / Item 2 (Properties) | Regional breakdowns are in Business or Properties |
| Debt maturity schedule | Item 8 Notes (debt note) | Maturities table is always in the long-term debt footnote |
| Lease expirations | Item 2 / Item 8 Notes | Lease expiration schedules appear in Properties or footnotes |

The chunker selects the relevant section(s) by name match — no LLM involved in routing. If a section is still very large (e.g., Item 8 with all footnotes), it does a secondary split by sub-headings (`<h3>`, bold paragraph openers) and keyword-scores the sub-chunks to pick the most relevant one.

### Step 4 — Token guardrail

Each chunk passed to the LLM is capped at ~25,000 tokens (~18k words). For context: a typical Item 1 section is 3–8k tokens. Item 8 notes can be 40k+, which is why the secondary split matters for debt and lease extraction.

---

## The 4 LLM calls

Each call receives: the targeted section text + a structured prompt. Every prompt requires the model to return:
1. The extracted data in a defined schema
2. A **verbatim quote** from the text it drew the number from (this is the citation)
3. The section it found it in

No citation → the UI shows an error state for that card, not a bare number. This is the hallucination guard.

## UI behavior

- Blank state: URL input centered, tagline, clean
- Loading: skeleton cards appear immediately, fill in one by one as SSE events arrive
- Done: 4 cards, each with extracted data + indented muted "Source:" block
- Buttons: "Copy summary" (plain text to clipboard) + "Export PDF" (jsPDF, one page, all 4 results)

**Key decision points I want your sign-off on before coding:**

1. **SSE vs. 4 parallel client requests** — I recommend SSE (single fetch, shared HTML parse). Happy to do parallel client calls if you prefer.
2. **`node-html-parser` vs. `cheerio`** — `node-html-parser` is faster and has zero native deps (important for Vercel edge). I'll use it unless you have a reason to prefer cheerio.
3. **Sequential vs. parallel LLM calls server-side** — Sequential is simpler and produces the "one card at a time" effect naturally. Parallel would be faster total but requires coordinating the SSE stream. Recommendation: sequential.

  export interface AnchorSpec {
    startText: string
    stopText: string
    maxChars?: number
  }
  export function extractByAnchor(cleanText: string, spec: AnchorSpec): string {
    const start = cleanText.indexOf(spec.startText)
    if (start === -1) return ""
    const searchFrom = start + spec.startText.length
    const stop = cleanText.indexOf(spec.stopText, searchFrom)
    const end = stop === -1
      ? Math.min(cleanText.length, start + (spec.maxChars ?? 50_000))
      : stop
    return cleanText.slice(start, end)
  }

4. Errors and Fixes:

- **Tenant concentration showing "Could not locate"** — new prompts requesting 25 tenantRows + full citation + tableRef exceeded 2048 token limit, truncating JSON so `parseLLMResponse` returned null. Fix: raised max_tokens. User said "don't put max on tokens" → raised to 8192 (model maximum, effectively uncapped while satisfying SDK requirement).

- **529 overloaded_error** — `{ "type": "overloaded_error" }` from Anthropic API. Added retry loop with `2^attempt * 1000ms` backoff, up to 4 retries, only for status 529 or 503. User confirmed it was platform-wide and not app-specific.

- **Geo table column misalignment** — CSS grid with each row as its own `grid` container means column widths are computed independently per row. Values like `$78.2B` vs `$90.7B` have different widths causing misalignment. Fix: converted entire L1 and L2 sections to HTML `<table>` elements with shared `<thead>`.

- **Dynamic Tailwind class bug** — `grid-cols-${hasConsol ? "[1fr_auto_auto_auto_auto]" : "[1fr_auto_auto_auto]"}` was silently dropped by Tailwind's content purger since the class string is not statically present. This was the previous L2 grid approach. Fixed by switching to `<table>`.

- **`loadEnv` Vercel build failure** — `vitest.config.ts` imported `loadEnv` from `"vitest/config"` which doesn't export it. Error: `Module '"vitest/config"' has no exported member 'loadEnv'`. Fix: import from `"vite"` instead.

- **Tenant bars missing after table conversion** — when TenantCard was converted from HBar components to a `<table>`, the progress bars were not included. Fix: added inline bar inside the `% NER` cell as a flex div with `w-14 h-1.5 bg-slate-100 rounded-full` + blue fill.

- **Donut chart gap** — `flex-1` on the table wrapper div caused the table to expand to fill card height, pushing donut to bottom. Fix: removed `flex-1` from table container, added `<div className="flex-1" />` spacer between donut and SourceBlock.

- **jsPDF removed from bundle** — `handleExportPDF` was replaced with `window.print()`, removing the dynamic `import("jspdf")`. Bundle size decreased.

5. Problem Solving:

- **Anchor disambiguation** — TOC uses "Geographic Distribution" (mixed case), body uses "GEOGRAPHIC DISTRIBUTION" (all caps); "Customers" appears as standalone `\nCustomers\n\n` in body only. All anchors verified unique via anchor-nav tests.

- **Debt table figures** — Exact figures (in thousands) are in Item 8 financial notes under "Long-Term Debt Maturities", not Item 7A which has rounded millions. Anchor correctly targets the exact-figure section.

- **Filing date not in HTML body** — Not available via DEI XBRL. Solution: call EDGAR submissions API by CIK + accession number.

- **L2 % calculation consistency** — LLM-returned regional totals may vary run-to-run. Solved by hardcoding `OM_REGIONAL_TOTAL` map and dividing child.gbvM by the constant, not by summed children.

- **Donut segment ordering** — `stroke-dashoffset = -cumArcLen` (negative, before increment) with `rotate(-90deg)` on each circle. Segments draw clockwise from 12 o'clock. Labels at `cx + r * cos(midAngle)`, `cy + r * sin(midAngle)` where `midAngle` accumulates from `-π/2`.
